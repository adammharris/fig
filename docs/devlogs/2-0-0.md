# `fig` version 2.0.0 "Sierra"

> "Flavor profile: Light-colored skin with a fresh, sweet flavor like a Riesling."
>
> — California Figs

`fig` Sierra is a big release! I initially planned this as multiple point releases, but then I realized `fig` could go farther.

## Fig

The star of this release is the new configuration format, eponymously named the Fig Language.

Why did I make a new format? Haven't I ever seen [XKCD 927](https://xkcd.com/927/)? To answer simply, I saw a gap in configuration languages that needed filling.

Some necessary context: there are a few ways to represent information depth with plain text. You could do YAML-style, with indentation (more indentation = more deep):

```yaml
this:
  is:
    a:
      very:
        deeply:
          nested: value
```

JSON-style, container markers (more opening markers = more deep):

```json
{"this":{"is":{"a":{"very":{"deeply":{"nested":"value"}}}}}}
```

TOML-style, sectionizing (header indicates depth):

```toml
[this.is.a.very.deeply]
nested = "value"
```

There are a few others that aren't practical for configuration, such as declaring a specific depth beforehand (so that you must change the depth number regularly). But there was one I saw could be useful: **repeated-marker**. Here is an example from fig:

```fig
this
> is
> > a
> > > very
> > > > deeply
> > > > > nested = value

# Also valid:
this
>is
>>a
>>>very
>>>>deeply
>>>>>nested=value

# alternatively...
this.is.a.very.deeply.nested = value

# alternatively, again...
this.is.a.very.deeply
> nested = value
```

Rather than relying on indentation, Fig relies on explicitly marking depth with the `>` character.

This solves a lot of problems with YAML, because YAML doesn't require a specific indentation level: only that the next indentation level is more than the previous. (Which causes confusion when indentation is)

Being able to represent depth in this way also solves TOML's issue of relying entirely on headers to represent depth at all. There is a simple escape hatch for depth when you need it without having to write JSON-like containers.

That said, Fig also supports that kind of syntax; it is very useful for small containers:

```fig
inline = { hello = world, seq = [ 1, 2, 3, ]}
```

Fig also supports literal JSON syntax for inline containers, so it is easy to copy/paste structured data into your Fig document:

```fig
json = { "hello": "world", "seq": [ 1, 2, 3 ]}
```

Fig is not, however, a literal superset of JSON—`fig get file.json -i fig` doesn't work. (`fig` supports JSON anyway.)

The core tradeoff Fig makes is: many ways to author the same data. This has the risk of making Fig files confusing, so a lot of work has been put into tooling to mitigate this: `fig fmt` works instantly, can be diff checked with `--diff`, and results in a clean, handwritten-looking file that preserves comments.

Here is a kitchen-sink example:

```fig
# === Containers, assignments, prefix-count depth ===
database                        # container header (bare word, no `=`)
> host = localhost              # database.host
> port = 5432                   # database.port  (bare number)
> pool                          # nested container header
> > size = 10                   # database.pool.size
> > timeout = 30                # database.pool.timeout

# === Dotted-key flattener (flatten within one line) ===
cache
> redis.host = 127.0.0.1        # cache.redis.host  (IP → string, 3 dots)
> redis.port = 6379             # cache.redis.port

# === Root-level assignments (zero markers, zero indent) ===
version = 2                     # a plain key in the root map
title = My Application          # bare string, spaces and all

# === Explicit typing:  key: type = value  (optional) ===
server
> port: int = 8080              # annotated (vs inferred)
> name: string = My Server      # bare string with spaces
> mode: enum = creative         # enum atom (explicit-typing-only)
> debug: bool = false

# === Values: literal-else-string ===
values
> answer = 42                  # int
> ratio = 3.14                 # float
> mask = 0xFF                  # hex int lexeme
> big = 1.5e3                  # float with exponent
> enabled = true               # bool (lowercase only)
> missing = null               # null
> flag = "true"                # STRING — quotes override the bool literal
> movie = 12 monkeys           # STRING — "12" is a number, not "12 monkeys"
> zip = 007                    # STRING — leading zero != number; padding kept
> team = "99"                  # STRING — quote to override the int literal
> when = 2026-07-01T12:00:00Z  # datetime (RFC-3339, self-identifying)
> day = 2026-07-01             # date (RFC-3339 full-date)
> huge: float = inf            # non-finite: explicit-typing-only
> nope: float = nan            # bare inf/nan would be plain strings

# === Quotes:  '' raw/literal   ""  escaped ===
strings
> raw = C:\Users\me\no-escapes'    # single quotes: backslashes stay literal
> esc = "line1\nline2\ttabbed"      # double quotes: \n \t \uXXXX honored
> snowman = "frozen ☃"
> "my.key" = present                # key with a literal dot MUST be quoted

# === Comment marker (#) only after whitespace — URLs survive ===
links
> docs = https://example.com/docs       # comment starts here; value ends before it
> api = https://example.com/v1#stable   # #stable has no leading space -> kept


# === Sequences — form A2 (`*` is an anonymous positional key) ===
# The element `*` sits in key position after the run: `> *` (normalized), `>*`
# (glued) — both appear below on purpose. YAML's `-` is a hard error.
servers
> *                            # first element (a map): fields one level deeper
> > host = a.com
> > port = 25565
> > backends                   # nested list field
> > > *                        # spaced form (normalized)
> > > > url = x
> > > > weight = 1
> > >*                         # glued form — parses identically to `> > >  *`
> > > > url = y
> *                            # second element
> > host = b.com

ports                          # scalar elements
> * 25565
> * 25566
> * 25567

# === Dotted section header — re-anchors baseline (like TOML [a.b.c]) ===
services.web.frontend          # selects/creates services.web.frontend
> replicas = 3                 # services.web.frontend.replicas (depth is RELATIVE)
> image = nginx:latest         # ':' in a value is fine -> bare string


# === Appending sequence headers — primary surface for nested lists of maps ===
# `a.b[]` appends an element to sequence a.b and re-anchors; fields stay 1 deep
# no matter how deep the LIST nests. Edit-stable (append, not index). A `+`
# line re-runs the most recent zero-marker `[]` header without retyping it.
jobs.test
> runs-on = ubuntu-latest
jobs.test.steps[]              # append a step, re-anchor here
> uses = actions/setup-node@v4
> with.node-version = 20
+                               # same as repeating `jobs.test.steps[]`
> run = npm test

# Nested lists of lists: a non-final `[]` means "the last element" — only the
# final `[]` in the path appends. So `spec.containers[].ports[]` appends a
# port to the LAST container, and `+` re-runs the whole path.
spec.containers[]              # append a container
> name = app
spec.containers[].ports[]      # last container -> append a port
> containerPort = 80
+                               # same path again -> another port, same container
> containerPort = 443
spec.containers[]              # a second container
> name = sidecar

# === Flow mode — value starting with [ or { (fig-inline OR pasted JSON) ===
flow
> tags = [a, b, c] # fig-inline: bare strings
> audience = [friends, Adam Harris, Makena Harris] # bare values keep spaces
> point = { x = 1, y = 2 }         # fig-inline object: `=` pairs, bare keys
> pasted = { "x": 1, "y": [2, 3] } # JSON object: `:` pairs, quoted keys

# === Quote-avoidance ===
dialogue
> * He wondered: "Well, what if I want commas in my string?"
> * She smiled: "Yes, of course you can have commas without escapes!"
> *: string = "You can even start with non-escaped quotes, if you specify the string type!"

# === Multiline strings — two flavors ===
docs
> license = '''       # '' raw/verbatim: no escapes, whitespace exact. First line skipped.
Copyright (c) 2026
All rights reserved. \n is literal here.
'''
> banner = """                # "" escaped + smart-dedent
      Welcome!
      Enjoy your ☃ stay.
      """
```

I plan to use Fig in my personal journal, and also provide support for it in my Diaryx project. I also hope it will help people who are frustrated with existing configuration languages. The `fig` project uses `.figl` files for configuration: these are converted to YAML and ZON on pre-commit for Github Actions and `build.zig.zon`.

## Breaking ABI changes:

- **Renamed the `fig_fm_*` frontmatter C ABI to `fig_embed_*`** and generalized it to any embedded region (frontmatter, endmatter). The old `fig_fm_*` symbols are removed.
- **`FigRegion` is now size-versioned**: set `out_region->size = sizeof(FigRegion)` before `fig_embed_extract`. Gained a `body` span (the host text outside the fences). Future fields can be appended without another major bump.
- **Embed API is fully generic over `EmbedType`** — no operation is tied to "frontmatter". The frontmatter-named convenience sugar is gone: in the Rust and TypeScript bindings, `Embed::frontmatter(md)` is replaced by `Embed::open(md, EmbedType::FrontmatterYaml)`, and the free function `split_frontmatter(content)` by `split(content, kind)`. The Rust `Extracted::frontmatter()` accessor is renamed to `content()`.
- **Rust**: Removed `serde`, `xml`, and `zon` from default features.
- **Rust**: changed signature of `replace_value`, `insert_value`, `set_value`, `append_value`, `prepend_value`, `set_sequence`, etc. from `items: &[Value]` to `items: impl IntoIternator<Item = impl Into<Value>>` (and similar) to reduce `Into` boilerplate required by consumers.

## New

- The Fig Language! I have chosen to stylize `fig`, the library, as inline code/raw text, because it is intended for use by programmers only. Fig, the language, is stylized as capitalized normal text—not like TOML, YAML, or JSON, because it is not an acronym or initialism, but simply an abbreviation of "configure." The file-extension is `.figl`, though the CLI also accepts `.fig` (conflict with Figma's native file format).
- `fig_editor_set` / `fig_embed_set` (and `Editor.set` / `Embed.set` in the bindings): upsert a mapping value — replace at the path, or insert the trailing key when only it is absent.
- `fig_embed_open_or_init` (and `Embed::open_or_init` / `Embed.openOrInit`): open an embedded region, or create an empty one — placed per the archetype (frontmatter at the top, endmatter at the bottom) — when the host has none, so the first `set`/`insert` lands the opening entry without a separate reserialize. Generic over `EmbedType`.
- **CLI**: `fig set <file> <path> <value>` — upsert (replace, or create the key when absent); on a host document with `--embed`, it creates the block too (open-or-init). `--seq <item>...` reconciles a sequence in place, preserving comments on survivors. `--embed <frontmatter|frontmatter-json|endmatter>` makes the embed archetype explicit (not just the `.md` frontmatter default) for both `set` and `get`. `fig get --body` prints the host prose outside the fences.
- **CLI**: `fig check` command for validating config files.
- Support for reading comments attached to nodes programmatically
- Update CLI with insert and delete operations
- `EmbedType::FrontmatterFig` (```` ```fig ```` fenced frontmatter) is now editable through `fig_embed_*`/`Embed`
- **YAML**: Support for YAML 1.1 (in addition to 1.2.2; 1.2.2 is still the default)
- **Typescript**: bindings have been revamped for usability.
- **ZON**: is now capable of editing, bringing it to feature parity with the other formats.
- **XML**: is not compiled in by default, but it now has a printer! (Use `-Dxml=true`)
- **Internal**: Versioning policy with `zig build check`, along with a host of build commands that compose the check command.
- **Homebrew**: Homebrew package now supports Linux; macOS binaries are split instead of universal.
- **GitHub Releases**: now releases macos-arm, macos-x84, linux-arm64, linux-x84, and windows binaries
